
# AgentSupport

## Инструкция по развертыванию

### **Development**:
```bash
docker compose -f docker-compose.yml -f docker-compose.dev.yml up --build -d
```

### **Production**:
```bash
docker compose -f docker-compose.yml -f docker-compose.prod.yml up --build -d
```

### **1 раз создание контейнеров tools, но НЕ запускать**:
```bash
docker compose --profile tools up --no-start db_seed db_cleanup_all db_clean
```

### **Запуск сидера (каждый запуск добавит еще записей)**:
```bash
docker compose start db_seed
```

### **Повторить еще раз**:
```bash
docker compose start db_seed
```

### **Очистка**:
```bash
docker compose start db_cleanup_all
```

## **Путь пользователя: Оператор техподдержки**

### 1) **Авторизация в системе**:
- Оператор техподдержки заходит в систему, используя свой корпоративный аккаунт для аутентификации (например, через SSO или учетную запись компании).

### 2) **Получение нового письма**:
- Система автоматически получает новые письма с корпоративной почты с помощью интеграции с почтовым API (например, SMTP).
- Новое письмо отображается в списке входящих запросов в интерфейсе оператора. В системе сразу начинается парсинг письма с извлечением ключевой информации (ФИО, контакты, номера приборов, суть запроса и т.д.).

### 3) **Обработка письма**:
- Оператор просматривает письмо и автоматически выделенные системой данные.
- Если информация из письма недостаточна или неправильно интерпретирована, оператор может вручную дополнить или отредактировать данные.
- В случае необходимости, оператор может отметить письмо как "Нуждается в дополнительной проверке", чтобы оно было передано на дальнейшую обработку специалисту.

### 4) **Классификация запроса**:
- Система автоматически классифицирует письмо по категориям (например, "неисправность", "калибровка", "запрос документации").
- Оператор может вручную скорректировать категорию запроса, если система не сделала это правильно.

### 5) **Генерация ответа**:
- На основе категории запроса и базы знаний система автоматически генерирует ответ, используя шаблоны или статьи из FAQ.
- Оператор проверяет предложенный ответ и в случае необходимости редактирует его (например, добавляет дополнительную информацию или уточняет детали).

### 6) **Отправка ответа**:
- После того как ответ готов, оператор может отправить его клиенту прямо через интерфейс системы.
- Ответ может быть отправлен через электронную почту или другие интегрированные каналы, такие как мессенджеры (например, Telegram или Slack).

### 7) **Просмотр таблицы и отчеты**:
- После отправки ответа оператор может перейти в раздел истории запросов и просмотреть таблицу с обработанными письмами.
- В таблице отображаются ключевые данные, такие как дата получения письма, категория запроса, статус обработки и время ответа.
- Оператор может фильтровать и сортировать запросы по различным критериям (например, по статусу, по категории запроса, по времени отклика).
- Таблица также предоставляет возможность скачать отчеты в формате CSV или XLSX для дальнейшего анализа.

### 8) **Завершение обработки**:
- Письмо помечается как "Обработано" или "Закрыто", если ответ был отправлен и запрос был решен.
- Оператор может вернуться к списку новых писем или продолжить работу с другими запросами, следуя аналогичному процессу.

## **Риски и минимизация**

| **Риск** | **Минимизация** |
| --- | --- |
| Ограниченное время на разработку | Распределить задачи на приоритетные и второстепенные, заранее подготовить каркас приложения. |
| Проблемы с интеграцией моделей AI | Использование предварительно обученных моделей, настройка базовой инфраструктуры до начала хакатона. |
| Проблемы с производительностью | Оптимизация кода, использование эффективных алгоритмов, настройка тестирования производительности. |
| Технические неполадки с инфраструктурой | Раннее развертывание и настройка окружения с использованием Docker Compose для быстрой настройки сервисов. |
| Ошибки в обработке данных и ответах | Регулярное тестирование системы, создание юнит-тестов для AI-агента. |
| Неудачный выбор технологий или инструментов | Тщательное исследование выбранных технологий до хакатона и использование их только в случае уверенности. |
| Нехватка опыта у команды с нужными инструментами или языками | Разделить задачи по уровням сложности, использовать простые и проверенные решения, провести обучение. |
| Неэффективная работа с данными (например, проблема с объемом данных) | Спроектировать эффективную схему базы данных, использовать кэширование, оптимизировать работу с данными. |
| Неоптимизированный процесс обработки запросов | Оптимизация алгоритмов обработки, использование очередей сообщений, тестирование времени отклика. |
| Проблемы с взаимодействием между фронтендом и бэкендом | Протестировать взаимодействие между компонентами, использовать Swagger для документирования API. |
| Ошибки при тестировании и валидации модели | Использовать набор тестовых данных, настроить систему для автоматического тестирования модели. |
| Конфликты в команде из-за распределения ролей и задач | Четко распределить роли на старте проекта, обеспечить постоянную коммуникацию между участниками команды. |
| Трудности с масштабируемостью | Сконцентрироваться на минимально жизнеспособном продукте (MVP), обеспечив возможность расширения в будущем. |
| Интеграция с внешними сервисами и API | Предварительная настройка и тестирование всех внешних интеграций, использование фальшивых данных для тестирования. |
